1. 概述

1.1 项目背景

嘉应学院每年举办大量学科竞赛，涵盖各学院、各专业方向。目前竞赛的报名、材料收集、评审组织等工作主要依赖线下手工方式（纸质表格、邮件、U盘拷贝等），存在以下痛点：

  • 信息分散，学生难以及时获取竞赛通知
  • 报名信息填写不规范，汇总统计耗时费力
  • 作品文件收集混乱，版本管理困难
  • 评审过程缺乏统一平台，评分记录不便追溯
  • 结果公示依赖群消息/公告栏，传播效率低

为此，拟开发一套学科竞赛报名评审系统，实现竞赛从创建、审核、发布、报名、作品提交、在线评审到结果公布的全流程数字化管理。

1.2 系统目标

  1. 统一管理：将竞赛信息、报名数据、作品文件、评审记录集中管理
  2. 流程规范：固化竞赛业务流程，减少人为疏漏
  3. 提升效率：自动化报名汇总、材料归档、分数计算等重复性工作
  4. 信息透明：学生可实时跟踪竞赛状态，评审结果可公开查询
  5. 数据安全：不同角色权限分离，保障数据安全与评审公正

1.3 适用范围

本系统适用于嘉应学院校内各级各类学科竞赛的报名与评审管理，包括但不限于：

  • 校级学科竞赛（教务处/团委主办）
  • 院级专业竞赛（各二级学院主办）
  • 省级/国家级竞赛的校内选拔赛
  • 创新创业类竞赛

1.4 术语定义

术语说明
竞赛/比赛由教师创建、管理员审核后发布的学科竞赛活动
团队由学生自行组建的参赛单位，包含队长和若干队员
个人参赛以个人身份报名，无需组队（可视为1人团队的特殊情况）
作品参赛者提交的成果文件，格式由建赛教师指定
评审评委对参赛作品进行打分评价的过程
决赛线下进行的最终评选环节（如答辩、展示等）

2. 角色与权限

2.1 角色定义

🛡️ 管理员 Admin

系统全局管理者，通常由教务处或相关职能部门工作人员担任。

  • 审核比赛：对教师提交的竞赛进行审核（通过/驳回）
  • 管理比赛：查看、编辑、下架、删除所有竞赛
  • 人员管理：管理系统用户账号（增删改查、角色分配）
  • 数据统计：查看全校竞赛数据报表
  • 系统配置：管理基础数据（学院列表、学科分类等）

👨‍🏫 教师 Teacher

竞赛的组织者和管理者，可同时担任不同竞赛的不同角色。

  • 创建比赛：填写比赛信息，提交管理员审核
  • 管理比赛：编辑自己创建的比赛信息（审核通过前可修改）
  • 评委指派：为比赛选择评审老师
  • 担任评委：对被指派的参赛作品进行评分
  • 担任指导老师：被学生关联为指导老师
  • 结果管理：确认并公布比赛最终结果

🎓 学生 Student

竞赛的参与主体，可浏览比赛、报名参赛、提交作品。

  • 浏览比赛：查看已发布的竞赛列表及详细信息
  • 报名参赛：以团队或个人形式报名
  • 创建/加入团队：组建参赛团队，邀请队员
  • 上传作品：在指定时间段内提交参赛作品
  • 查看结果：查看评分及获奖情况

4. 业务流程

4.1 竞赛全生命周期流程

主流程：创建竞赛 → 提交审核 → 管理员审核 → 发布竞赛 → 学生报名 → 提交作品 → 评委评审 → 确定决赛名单 → 线下决赛 → 公布结果

以下对各环节进行细化说明：

4.1.1 创建竞赛

  1. 教师登录系统，进入"竞赛管理"，点击"创建竞赛"
  2. 填写竞赛基本信息（名称、主办单位、学科分类、封面、简介）
  3. 设置时间节点：报名开始/截止、作品上传开始/截止、评审开始/截止、决赛时间地点、结果公布时间
  4. 系统校验时间逻辑：报名开始 < 报名截止 < 作品上传截止 < 评审开始 < 评审截止 < 决赛时间 < 结果公布时间
  5. 配置参赛规则：允许格式、文件大小上限、是否允许个人参赛、团队人数范围、是否需要指导老师
  6. 填写详细说明（富文本编辑器，支持图文排版）
  7. 保存草稿或提交审核

4.1.2 审核与发布

  1. 管理员在"审核管理"中查看待审核竞赛列表
  2. 点击进入竞赛详情，逐项审查信息完整性与合规性
  3. 审核通过：竞赛状态变为"已发布"，在竞赛广场对学生可见
  4. 审核驳回：填写驳回理由，竞赛退回至建赛教师的草稿列表
  5. 教师根据驳回理由修改后重新提交，重复审核流程

4.1.3 学生报名

  1. 学生在竞赛广场浏览已发布竞赛，按条件筛选
  2. 点击目标竞赛查看详情，确认参赛意愿
  3. 团队报名：
    • 创建团队（或使用已有团队）
    • 邀请队员加入，队员确认
    • 填写报名表单（作品名称、指导老师等）
    • 提交报名
  4. 个人报名：
    • 直接填写报名表单
    • 提交报名
  5. 报名截止后，系统锁定报名名单，生成报名汇总

4.1.4 作品上传

  1. 在作品上传时间段内，队长登录系统
  2. 进入"我的报名"→ 目标竞赛 → "作品管理"
  3. 上传作品文件（支持多个，系统校验格式和大小）
  4. 可删除、重新上传，以最后一次提交为准
  5. 点击"确认提交"锁定作品，或等待截止时间系统自动锁定

4.1.5 在线评审

  1. 建赛教师在评审开始前完成：
    • 设置评分标准（维度、分值、权重）
    • 指派评委（从教师列表中选择）
    • 配置分配方式（全体评审 / 分组评审）
  2. 评审开始后，评委在"我的评审"中看到待评作品
  3. 评委下载/查看作品，逐项打分并填写评语
  4. 评委可暂存评分草稿，确认后正式提交
  5. 建赛教师实时查看评审进度
  6. 评审截止后，系统自动汇总计算

4.1.6 决赛名单与结果公布

  1. 建赛教师查看评分汇总及排名
  2. 根据预设名额确定决赛入围名单
  3. 公布决赛名单（系统通知入围团队）
  4. 线下决赛结束后，建赛教师录入决赛结果或调整最终排名
  5. 设置奖项等级，确认获奖名单
  6. 点击"公布结果"，系统生成结果公告并公开发布

4.2 异常流程处理

异常场景处理方式
报名截止时无报名系统自动将竞赛标记为"报名不足"，通知建赛教师，教师可选择延期或取消
作品上传截止时有报名但未上传作品视为放弃参赛，自动取消该报名资格
评委人数不足评审开始前，建赛教师需补充评委；评审开始后仍不足则提示并记录
评审时间到期仍有评委未完成评分仅以已提交的评分计算平均分，未评分的评委不计入（或建赛教师可手动延期）
决赛时间/地点变更建赛教师可编辑决赛信息，系统自动通知相关参赛者
竞赛需要取消管理员可强制下架竞赛，并填写取消原因，系统通知所有已报名学生

7. 非功能需求

7.1 性能要求

指标要求
页面加载时间首屏加载 ≤ 3秒（P95）
接口响应时间普通查询 ≤ 500ms，复杂统计 ≤ 3s
文件上传支持最大 500MB 单文件上传，支持断点续传
并发用户数支持峰值 2000 并发用户
数据库查询热点数据（竞赛列表）需加缓存

7.2 安全要求

  • 对接学校统一身份认证，不独立存储密码
  • 敏感操作（审核、公布结果）需二次确认
  • 文件上传需校验类型白名单，防止恶意文件上传
  • 接口需做权限校验，防止水平越权（如学生调用教师接口）
  • SQL注入、XSS、CSRF 等常见Web漏洞防护
  • 评审期间，评委之间不可互看评分（除非建赛教师开放）
  • 操作日志记录，满足审计要求

7.3 可用性要求

  • 系统可用性 ≥ 99.5%（年度）
  • 关键时间节点（报名截止前、评审截止前）需保证系统稳定
  • 数据每日自动备份，保留最近30天

7.4 兼容性要求

  • 支持 Chrome、Edge、Firefox 最新两个大版本
  • 支持移动端响应式布局（至少保证报名和查看结果可正常操作）
  • 支持校内网络环境访问（校园网 + VPN）

7.5 技术栈建议

层级建议技术备注
前端Vue 3 + Element Plus / React + Ant Design按团队技术栈选型
后端Spring Boot / Django / Node.js (NestJS)按团队技术栈选型
数据库MySQL 8.0+ / PostgreSQL 14+
缓存Redis会话管理 + 热点缓存
文件存储MinIO / 阿里云OSS / 本地存储按IT基础设施选择
消息队列RabbitMQ / Redis Stream异步通知、文件处理
部署Docker + Nginx
